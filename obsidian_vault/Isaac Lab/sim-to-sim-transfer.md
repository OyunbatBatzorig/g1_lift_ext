# Sim-to-Sim Transfer: IsaacLab → MuJoCo

## Locomotion (solved)
- **Policy export format:** TorchScript (`.pt`) via `export_policy_as_jit` in `scripts/rsl_rl/play.py` (RSL-RL exports both jit and ONNX, but `sim2sim.py` only consumes the jit one). Loaded in MuJoCo with a plain `torch.jit.load(...)` — no normalizer object needed separately since it's traced into the exported module. Alongside the policy, `play.py`'s export also dumps the full resolved training `env.yaml`, which `sim2sim.py` re-parses at runtime (`sim2sim_helper.parse_training_env_yaml`) to pull PD gains, armature, effort limits, default joint angles, obs scales, action scale, `sim.dt`, and `decimation` — i.e. the MuJoCo side never hardcodes training hyperparameters, it derives them from the same config IsaacLab produced.
- **Observation space mapping** (IsaacLab order → MuJoCo order): IsaacLab's joint order for the exported policy is captured by hand into `g1_lab.yaml` as `ordered_joints_lab` (a flat list of 37 joint names, "found in the log"). `sim2sim_helper.indices_mapping_lab_to_mjc` builds a permutation `joint_ids` by looking up each lab-ordered joint name's index in the MuJoCo model's joint list (`get_sorted_mjc_joint_names`, which just walks `m.njnt` skipping index 0, the floating base). PD gains/effort limits/armature/default angles are stored in MuJoCo joint order; `qj[joint_ids]` / `dqj[joint_ids]` reorder MuJoCo state into lab order to build the obs vector, and `target_dof_pos[joint_ids] = target_dof_pos_lab` reorders the policy's lab-order action back into MuJoCo order before it's applied. The obs vector itself is hand-assembled to match the training `ObservationsCfg` term order exactly: `[base_lin_vel(3), base_ang_vel(3), projected_gravity(3), velocity_commands(3), joint_pos_rel(37), joint_vel_rel(37), last_action(37)]` = 123, matching `num_obs` in the config.
- **Action space / control mode differences:** Training used `JointPositionAction` (scale 0.5, offset = default joint pos, going through IsaacLab's `ImplicitActuator`, which resolves position targets to torque internally via the physics engine). MuJoCo has no equivalent implicit actuator, so `sim2sim.py` reimplements it explicitly: `tau = kp*(target_q - q) + kd*(-dq)`, clipped to `effort_limit`, and written straight to `d.ctrl`. This only works because `g1_37_lab.xml` declares plain `<motor>` actuators (direct torque, no built-in position servo) — if the XML had used MuJoCo position actuators instead, the PD would have been applied twice. kp/kd/effort_limit/armature are all pulled from the actuator groups in `env.yaml` via regex pattern matching against joint names (`get_matching_value`), not hardcoded per robot.
- **Control frequency matching:** `m.opt.timestep` is set directly from `sim["dt"]` (0.005s → 200 Hz physics) and the policy is only queried every `control_decimation = sim["decimation"]` steps (4 → 50 Hz control), both read straight out of the training `env.yaml` rather than re-entered by hand — so frequency mismatches are structurally avoided as long as the exported `env.yaml` is trustworthy.
- **Domain randomization handling** (on in IsaacLab, off/adjusted in MuJoCo?): Not replicated in MuJoCo at all, and it doesn't need to be — DR (material randomization, `reset_root_state_uniform`, `reset_joints_by_scale`, per-term observation noise) is a training-time robustness technique baked into the policy weights. The captured `env.yaml` in this commit even shows `observations.policy.enable_corruption: false`, confirming it was exported from a play/eval-mode run where the noise models are defined but gated off, not from a live-training config with corruption still on.
- **Validation approach** (how you confirmed the transfer worked): Purely visual — `mujoco.viewer.launch_passive(m, d)` opens an interactive viewer, the sim runs for `simulation_duration` (10s) wall-clock with a fixed `cmd_init` velocity command (`[0.0, 0.5, 0.5]`, never updated during the run), and success is "does it walk without falling over." No automated comparison against IsaacLab rollouts, no logging/plotting of joint trajectories, no keyboard/joystick control to vary the command mid-run.
- **Gotchas / what broke first:**
  - `obs[:3] = base_vel` uses `d.qvel[:3]` directly. For a MuJoCo free joint, `qvel[0:3]` is the base linear velocity in the **world** frame, but IsaacLab's `base_lin_vel` observation is body-frame. Unlike `get_gravity_orientation`, which explicitly rotates by the quaternion, there's no corresponding rotation for linear velocity here — a frame mismatch that's silently tolerated (small when roughly upright) rather than fixed. Worth checking if this was ever corrected.
  - `sim2sim_helper.py`'s actuator-parsing loop sets `joint_name = pattern` and then checks `if re.match(pattern, joint_name)`, which is tautologically always true — the code even leaves in a `# TODO: Remove this condition if not needed` comment. Dead logic shipped as-is.
  - The whole approach was short-lived: this commit (2025-06-13) was superseded 17 days later by `33d3ba2` ("update to main", 2025-06-30), which **deleted** `depoly/sim2sim.py`, `sim2sim_helper.py`, both example env/XML bundles, and `doc/sim2sim.gif` outright, replacing them with a C++ `deploy/` stack (`FSM/State_RLBase.h` etc., IsaacLab-mirroring headers under `deploy/include/isaaclab/`) aimed at real-hardware sim2real over DDS rather than MuJoCo sim2sim. So the pure-Python MuJoCo verification path documented above never matured further in-tree — it was a one-shot proof of concept before the project pivoted to C++ deployment. (`d310472`, same day, also just fixes the `depoly` → `deploy` directory-name typo along the way.)
  - Fixed velocity command only — no path from this commit to interactively changing `cmd` during a run; that capability (`support_keyboard_velocity_command`) only appears later, in the C++ deploy stack.
- **Source commit(s):** `f81916c` "sim2sim verification in mujoco" (2025-06-13, adds `depoly/sim2sim.py` + `sim2sim_helper.py` + `g1_lab.yaml` + example G1-37dof policy/env/XML bundle), `42f1850`/`e1b41ec` (2025-06-16, adds a second G1-23dof example + placeholder dirs), `d310472` (2025-06-30, `depoly`→`deploy` rename), `33d3ba2` (2025-06-30, "update to main" — **removes** the MuJoCo sim2sim assets, replaced by C++ sim2real deploy code).
